Part I: Refactoring & Polishing Test Suite

Before you begin refactoring, consider your test suite from Assignment 1.1. Did your moderator, the TAs, or your peers from discussion section suggest ways to improve the coverage of your test suite? Yes, yes, you should have written your tests before implementing your functionality last week, but if for whatever reason, your test suite could be more thorough, spend some time enriching your test suite before you begin refactoring. Doing so will help you refactor more quickly, and to be more confident in the correctness of your refactorings.

If you have not already done so, consider using a code coverage tool such as EclEmma for Eclipse, or the integrated code coverage features in IntelliJ IDEA to quantify how thorough your test suite truly is.

You should refactor your code to eliminate any code smells (e.g. use communicative naming, decompose larger methods into smaller separate methods, etc), add missing tests, or other problems discussed in section. If you haven't already, you should be using a priority queue and a hashtable for the open and closed sets, respectively. Look here for discussion about these (as well as more advanced data structures) in improving the performance of A*.

Part II: Mock Unit Testing

For this assignment, we will be taking advantage of a mock unit test framework called Mockito. If you are unfamiliar with the concept of mock unit testing, fear not: it's a very simple concept.

What is Mock Unit Testing?

With a traditional unit testing framework like jUnit, setup some test case, interact with the module being tested, and verify the correctness of the output. Put simply, each unit test should follow three basic steps:

  1. Setup (where we setup the test case)
  2. Test (where we actually interact with the module/class we are testing)
  3. Verify (where we use assertEquals or something similar to test the output for our module/class)

At a high level, mock unit testing simply allows us to test that a function (or class, object, or whatever) is used as we expect. Specifically, such frameworks allow us to remove the implementation of a particular function at runtime, so that we can test that it is called without actually running its implementation.

To supplement our jUnit test suite used thus far, we will be using Mockito, a mock unit testing framework for Java that plays well with jUnit. Mockito is extremely powerful, yet has a dead simple API.

Why Use It?

The primary benefit of mock unit testing is to allow you to test components of your code in isolation, which yields a twofold benefit:

1. First, when a method is called on a mock object, the true implementation for that object is not executed. For instance, in our example above, since we only ever use the mockedList object, the true implementation of the List class is never executed (i.e. the internal data structures for the List, etc). In short, each method call to a mocked object does nothing.
2. Second, the mock object records when its methods are called, the parameters to those methods, the number of times they were called, and even the order in which different functions were called. Thus, it allows us to test that the mock object is being used correctly by some other component.

The power of mock unit test frameworks is displayed more clearly in situations where the implementation of a particular function or module should not be run during the run of the project's unit test suite. For example, consider the following example scenario.

You are developing an application that interacts with a database (and that database contains important data!). You wouldn't want to touch the database each time the test suite is run, because you may lose data, slow down the test suite's execution, or perhaps because you believe doing so in the test suite is just downright inelegant. In this case, you could create a module responsible for interacting with the database, and exclusively use mock objects of this module. Thus during unit tests, the database will not be accessed or modified, but we can still test that the proper calls are made to the database module.

Similar scenarios may arise, for example in the cases of networking (we don't want to touch the network every time we run the project's unit tests!), or disk I/O (for example image I/O as in this assignment).

If you're still not convinced that mocking is a good idea, please take a look at a few of these resources:
  • A concise explanation on Stack Overflow: http://stackoverflow.com/questions/1414032/why-create-mock-objects
  • A lengthy but readable blog post: http://codebetter.com/jeremymiller/2005/12/20/why-and-when-to-use-mock-objects/
  • A more involved example from IBM (using a different mock test framework): http://www.ibm.com/developerworks/library/j-mocktest/index.html

Your Task

For this assignment, you must download and add Mockito to your project, and use it to test that your mazes are being read and written as expected. For instance, you may want to test your maze running or solving components using a mock object for file I/O, so that the actual filesystem is never touched during the test execution.

Do not checkin the Mockito JAR file to SVN. This wastes space, and will result in a point deduction for Code Submission.

To install Mockito into your project, download the mockito jar file, and add it as a dependency to your project.

In Eclipse, this simply entails copying the JAR file into your project and adding it to the build path: http://www.eclipse.org/forums/index.php/t/346442/.

In IntellIJ, the processes is a few more steps, but is nonetheless fairly straightforward: http://stackoverflow.com/questions/7065402/how-to-add-external-library-in-intellij

Again, please do not check the JAR file you download into SVN!

Part III: Image I/O

As of now, your library should be able to handle arbitrary simple mazes, regardless of the input or output format of the maze. This week, we would like you to expand your library to be able to:

1. Parse mazes from a BMP image
2. Output the solved maze to a BMP image

You are required to solve each of these mazes using your library. Each is a grayscale BMP image, where the black pixels represent "road level" (passable) locations, and the white pixels represent the snow covered tops of (impassable) walls.

  • maze0.bmp
  • maze1.bmp
  • maze2.bmp

Likewise, your library is required to output the solved mazes for this week. Below are the corresponding solved mazes. For this week, we expect your solutions to look similar to these, but they do not need to be identical. The following are solved examples of the above mazes, given various starting/ending points:

  • maze0_sol1.bmp
  • maze0_sol2.bmp
  • maze1_sol1.bmp
  • maze1_sol2.bmp
  • maze2_sol1.bmp

Your implementation should use a variable to enable you to set the grey level threshold of what is considered an open space and what is considered a wall.

Image IO in Java

The basics of image IO in Java are simple if you know what classes to look for. Try the following code sample to open a BMP file

//Create file for the source
File input = new File("c:/temp/image.bmp");

//Read the file to a BufferedImage
// Surround this with try/catch or have your method
// throw an exception
BufferedImage image = ImageIO.read(input);

// Get width/height
int height = image.getHeight();
int width = image.getWidth();

Color color = new Color(image.getRGB(x,y));

Note that you will have to import some classes (Eclipse will tell you this, but so will I just for completeness):

import java.awt.image.BufferedImage;
import java.io.File;
import java.io.IOException;
import javax.imageio.ImageIO;
import java.awt.Color;

Part IV: Simple User Interface

Finally, we would like you to add a wrapper to your maze solving library to allow performing basic maze function via a text-based interface (i.e. usable from the terminal or console in your IDE of choice). It must allow the user to interact with your library via the console or command line, and we suggest (but do not require) that you make an interactive text-based interface for users.

Your API should expose all basic functionality for your maze library, specifically:

  • Read an input image and parse it into a maze
  • Test whether a maze is solvable
  • Allow the user to choose the heuristic to use for solving the maze
  • Allow the user to choose the pathfinding algorithm to use for solving the maze
  • Find a solution in the maze if possible
  • Write any solution to an output image showing the best path overlaid on the input maze image
Although it is not necessary for this week's assignment, you may want to study the MVC (Model View Controller) architecture. How easy would it be to replace this week's simple text-based user interface with a different user interface?
